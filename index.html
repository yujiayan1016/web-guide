Web 排版与设计基准完整规范

Mobile First、PC 非线性放松、DJI vs Kandao 实战对比

版本: 1.0 适用: 设计师 / 前端 / 产品

一、核心结论

Web 的唯一设计基准是 Mobile（375px CSS 视口）。

PC 端不是新的设计体系，而是：在 Mobile 规则成立的前提下，通过断点进行“响应式增强”。

PC 端不是 Mobile ×2

❌ 错误理解

PC 屏幕更大 → 字号 ×2

✅ 正确理解

PC 是“非线性放松比例”，而不是线性放大。

  • 正文：几乎不变
  • 标题：明显变大
  • Hero：才可能接近 ×2

二、为什么 PC 不能等比放大（感知层逻辑）

1️⃣ 人眼感知的是“视角”，不是 px

设备 常见观看距离
手机 25–35 cm
笔记本 45–60 cm
显示器 60–80 cm

屏幕越大，人离得越远，会自然抵消一部分“放大需求”。

2️⃣ 如果 PC 全部 ×2，会发生什么？

层级 Mobile ×2 后（错误） 合理 PC
正文 16 32 ❌ 16–18
模块标题 20 40 ❌ 24–30
页面标题 24 48 ⚠️ 30–36
Hero 28 56 ⚠️ 36–48

三、Web 的真实技术模型

1️⃣ Web 不是“画布缩放”，而是“规则重算”

Web 设计的本质是：定义规则，而不是定义结果尺寸。

🔴 核心差异演示：视口变大 (375px → 390px)

当屏幕变宽 15px 时，App 通常选择等比放大，而 Web 选择填充空间。

❌ App 逻辑 (Canvas Scaling)

375px (基准)
390px (放大)

整体等比放大，字号变大，
信息量未增加。

✅ Web 逻辑 (Fluid Reflow)

375px (基准)
390px (变宽)

Padding 不变，内容变宽。
单行容纳更多内容。

2️⃣ Mobile First 是 Web 的必然选择

原因：1. 最小视口约束最严格；2. 先解决信息密度；3. PC 只是“空间变大”。

如果 375 下逻辑不闭合，PC 一定会乱。

八、终极总结

Web 排版的本质不是“适配尺寸”，而是“在不同空间下保持阅读体验稳定”；
Mobile 是规则源头，PC 是非线性放松后的表现形态。
